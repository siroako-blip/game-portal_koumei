# devlog スキル

このプロジェクトの `DEVLOG.md`（開発の遷移＝何を・なぜ変えてきたかの年表）を扱う。
`DEVLOG.md` はリポジトリ直下にあり、**最新が上**（逆時系列）。

引数 `$ARGUMENTS` で動作を切り替える:

## 1. 追記モード（既定 / 引数が空・`add`・`記録`・`残す` 等）

直近の開発内容を1エントリにまとめ、`DEVLOG.md` の **いちばん上**（冒頭の説明とフォーマット例の直後、最初の `## YYYY-MM-DD` エントリの前）に追記する。

手順:
1. `DEVLOG.md` を読み、最新エントリの日付を確認する。
2. その日付以降の変更を把握する:
   - `git log --since=<最新エントリ日付> --format="%ad | %s" --date=short`
   - 未コミットの変更があれば `git status --short` / `git diff --stat` も確認。
   - 会話中で今まさに行った開発があれば、それを最優先で反映する。
3. **節目（方針転換・機能追加・大きなリファクタ）だけ**を拾い、細かな修正は1行にまとめるか省く。
4. 次のフォーマットで追記する（日付は実際の今日の日付）:
   ```
   ## YYYY-MM-DD — 見出し（その節目で何をしたか）
   - 変更点と **なぜ** そうしたか
   - （任意）見送った案・既知の課題・関連コミット
   ```
5. 追記後、エントリ内容をユーザーに短く報告する。

注意: 既存エントリは書き換えない。重複する節目があれば追記せず、その旨を伝える。

## 2. 参照モード（引数が `history`・`経緯`・質問文 等）

`DEVLOG.md` を読み、ユーザーの問い（例:「なぜマルチゲーム化したか」）に該当する遷移を要約して答える。
ファイル全文を貼り付けず、**問いに関係する部分だけ**を簡潔にまとめる（トークン節約）。

## 共通の注意

- `DEVLOG.md` が無ければ、作成してよいか確認する。
- コミットはユーザーが明示的に求めたときだけ行う。コミットする場合はメッセージ規約（`CLAUDE.md` 参照、`docs: ...` 形式）に従う。
